# Singularity

Singularity is a Roblox combat project built with Rojo, Wally, and Luau. The codebase is organized around shared client/server services, a React-based UI, and server-owned gameplay systems for inventory, skills, movement, damage, quests, rounds, and player data.

## Tooling

This project uses Rokit to pin Roblox development tools:

- Rojo `7.6.1`
- Wally `0.3.2`
- wally-package-types `1.6.2`

Install the tools with:

```bash
rokit install
```

Install Wally dependencies with:

```bash
wally install
```

`Packages/` and `ServerPackages/` are generated by Wally and are intentionally ignored by git. Keep `wally.toml` and `wally.lock` committed.

ByteNet is declared in `wally.toml` for the new network layer. Run `wally install` after changing dependencies so `Packages/ByteNet` and `wally.lock` stay in sync.

## Development

Start Rojo from the project root:

```bash
rojo serve default.project.json
```

Then connect Roblox Studio with the Rojo plugin.

Build a place file when needed:

```bash
rojo build default.project.json --output build.rbxl
```

Generated `.rbxl` and `.rbxlx` files are ignored because the source of truth is this repository.

## Project Layout

```text
src/
  client/                         Client startup scripts
  startup/MountUI.client.luau      Mounts the React UI
  features/app/rootUI.luau         Main React UI tree
  shared/                          ReplicatedStorage.Shared services
  server.server.luau               Server bootstrap
  server/Services/                 ServerScriptService.Services
```

Service folders follow this pattern:

```text
src/shared/[Name]Service/
  [Name]ServiceClient.luau
  [Name]ServiceUtils.luau

src/server/Services/[Name]Service/
  [Name]ServiceServer.luau
```

Skill logic is split into shared/client skill interfaces and server skill modules:

```text
src/shared/SkillService/ClientSkills/
src/server/Services/SkillService/Skills/
```

## Main Systems

- `WeaponRegistry` and `shared/Configs/Weapons` define weapon ids, combat defaults, UI metadata, and physical model names.
- `InventoryService` handles inventory state, equipped weapon/skill/skin attributes, weapon factory spawning, and weapon preview model publishing.
- `SkillService` owns skill activation and server-side skill behavior.
- `DamageCalcService` centralizes damage, hit feedback, hit particles, and damage direction events.
- `MovementService` owns movement/stamina state and publishes gameplay HUD attributes.
- `DataService` wraps player data through the Wally ProfileService package.
- `NetworkLayer` is the ByteNet entry point and currently keeps the old remotes safe until generated packages are refreshed.
- `rootUI.luau` contains the current Persona-inspired React UI for inventory, skill HUD, stamina, hit indicators, and damage direction indicators.

## Adding Weapons

Weapon data lives in Luau config modules, while physical assets live in `ServerStorage.WeaponInventory` as `Tool` instances. Services should read weapon data through `ReplicatedStorage.Shared.WeaponRegistry`, not from ModuleScripts inside the 3D model.

Add a config module first:

```text
src/shared/Configs/Weapons/FairyDagger.luau
```

Example:

```luau
--!strict

return {
	Id = "FairyDagger",
	DisplayName = "Fairy Dagger",
	ModelName = "FairyDagger",
	VisualName = "FairyDagger",
	Cost = 0,
	Color = Color3.fromRGB(255, 135, 230),
	Description = "Light dagger with fast recovery and precise close-range pressure.",
	BaseKnockback = 28,
	DamageMultiplier = 1.35,
	DamagePerHit = 18,
	HitCooldown = 0.35,
	PreviewRotation = CFrame.Angles(math.rad(-8), math.rad(-28), math.rad(38)),
}
```

Then add the id to `ORDER` in `src/shared/WeaponRegistry.luau`.

The physical tool can have any internal visual structure as long as it follows this runtime contract:

- The `Tool` name is the weapon id, such as `FairyDagger`.
- The tool has a `Handle` `BasePart`. This can be an invisible hitbox.
- The visible mesh/model can live anywhere under the tool.

If an imported asset starts as `Workspace.Handle.FairyDagger`, wrap it into a new `Tool` like this:

```text
ServerStorage
  WeaponInventory
    FairyDagger (Tool)
      Handle (Part, invisible hitbox)
        FairyDagger (Model, visible mesh)
      IdleAnim (Animation)
      SwingAnim (Animation)
```

`DataService`, `InventoryService`, and `rootUI` read the weapon list and basic metadata dynamically from `WeaponRegistry`.

## Notes

- Do not manually edit generated Wally package folders.
- Do not change `default.project.json` unless the Rojo tree needs to change.
- Prefer adding gameplay behavior through the existing service pattern so client and server logic stay easy to find.
